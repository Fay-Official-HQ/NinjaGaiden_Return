剑术输入手感优化需求

问题描述

当前剑术触发机制为：按下 L 键进入“剑术备战”状态，随后按下一个方向键（前、后、上、下）触发对应的剑术：

· L + 前 → 前冲
· L + 后 → 旋斩
· L + 上 → 上挑
· L + 下 → 下劈

实际测试中发现手感问题集中在 上挑 和 下劈，特别是当玩家在空中时：

· 玩家常常在跑动中跳跃（此时一直按着 前 方向键不放）
· 跳跃到空中后，玩家希望执行 上挑（或 下劈），于是按下 L 的同时手指尝试搓到 上（或 下）
· 但由于 前 键始终处于按下状态，系统在 L 按下的瞬间读取到的方向是 前，导致误触发 前冲，而不是玩家想要的上挑或下劈

同样的问题也会发生在：

· 玩家在空中想下劈，但依然按着 前 键，导致误放前冲
· 玩家在墙面或地面想上挑，但按住前跑动中，同样会误判

玩家希望：即使一直按住前，只要在按下 L 后的极短时间内快速按下上（或下），系统就应该识别为上挑（或下劈），而不是前冲。

而 L + 后（旋斩）因为需要按反方向，与跑动方向冲突，误触概率低，可以不特殊处理。

---

解决方案概述

引入一个 短时方向输入窗口（约 0.07~0.1 秒，对应 4~6 帧，60fps）。
L 键按下时，不立即执行剑术，而是启动一个计时器，在此窗口内持续监测方向输入，窗口结束后再根据 最终收集到的方向 决定触发哪个剑术。

窗口内方向收集规则：

· 每帧（或每次方向输入变化）更新当前“有效方向”
· 方向优先级：上/下 > 前/后。即只要窗口内出现过上或下，就以最后一次出现的上/下为准；如果从未出现过上/下，则以窗口结束时的前/后方向为准。
· 如果窗口内从未有任何方向输入，则触发默认行为（建议：无方向时不触发任何剑术，或者触发旋斩，由设计决定）。

这样，玩家一直按住前，然后按下 L 并快速向上滑动，窗口会先记录到“前”，但随后记录到“上”，最终触发上挑。整个过程延迟极短，玩家几乎无感知。

详细需求

1. 输入窗口参数

· 窗口长度：0.08 秒（约 5 帧，可调，建议作为 @export 变量）
· 计时器实现：使用 Timer 节点或基于 _process 的帧计数

2. 方向收集逻辑

· 方向输入源：Input Map 中的 move_left, move_right, move_up, move_down
· 将方向抽象为："left", "right", "up", "down"
· 当 L 按下时，清空上一次的缓冲方向，启动窗口。
· 在窗口期间，每当方向输入发生变化（包括新方向按下或原方向松开），更新一个临时变量 pending_dir。
· 窗口结束时，根据以下优先级决定最终剑术：
  1. 如果 pending_dir 为 "up" → 触发 上挑
  2. 否则如果 pending_dir 为 "down" → 触发 下劈
  3. 否则如果 pending_dir 为 "right" → 触发 前冲
  4. 否则如果 pending_dir 为 "left" → 触发 旋斩
  5. 否则（无方向）→ 不触发任何剑术，仅播放一个提示音效（可选）

注意：left 和 right 相对于玩家当前面朝方向。即：

· 面朝右时：right = 前，left = 后
· 面朝左时：right = 后，left = 前

因此，在判定前需要将原始左右方向根据 is_facing_right 转换为“前/后”逻辑，然后再匹配剑术。

3. 特殊边界情况处理

· 窗口内从未有方向输入：不触发剑术，避免空挥。但可播放一个“未就绪”音效提示。
· 窗口内先后输入了上和前：最终触发上（优先级高）
· 窗口内先后输入了下和前：最终触发下
· 窗口内输入了上然后松开，最后只有前：只要出现过上，最终就触发上（因为优先级高且最后一次方向为上）。这符合玩家“快速滑一下上”的意图。
· 窗口内只输入了前：触发前冲

4. 与其他系统的兼容性

· 本窗口机制只影响剑术触发，不应干扰角色的普通移动、跳跃、攻击等输入。
· 如果角色在窗口期间死亡或进入不能释放剑术的状态，应取消窗口并清除 pending_dir。
· 窗口期间如果玩家再次按下 L（快速连按），应该重置窗口，重新开始计时。

5. 性能与实现要求

· 使用 _unhandled_input 或 _input 捕获 L 键事件。
· 使用 Timer 节点或 Engine.get_process_delta_time() 累加帧计数实现窗口。
· 方向更新可以放在 _process 中每帧调用 get_vector 并更新 pending_dir，或者在方向键的 InputEvent 中实时更新。
· 提供 @export 变量 sword_input_window_time 以方便调试手感。

## 一、整体评价

- **核心机制**（短时方向窗口 + 上/下优先）抓住了症结，延迟极短，玩家无感。
- **参数可调**（@export）便于测试手感。

## 二、建议优化点

### 1. 窗口逻辑应放在 `SwordReadyState` 中
你现有的剑术触发流程是：`L键` → 进入 `SwordReadyState` → 方向键 → 对应剑术状态。  
将这个窗口机制**嵌入 `SwordReadyState`** 更合理：
- 按下 `L` 键时，状态机切换至 `SwordReadyState`。
- 在该状态的 `enter()` 中启动窗口计时器，并播放备战动画（可悬停）。
- 窗口期间每帧更新 `pending_dir`。
- 窗口结束后，根据方向调用 `state_machine.change_state(...)` 切换至对应剑术状态。
- 若窗口内无有效方向，则 `state_machine.change_state` 回到 `IdleState` 或 `RunState`，不触发剑术。

这样既不改动你庞大的状态机逻辑，又能用最小改动实现。

### 2. 方向收集的简化实现
不需要监听按键事件，只需每帧用 `Input` 单例读取当前按下的方向，并根据优先级记录：

```gdscript
func _update_pending_dir():
    var left = Input.is_action_pressed("nav_left")
    var right = Input.is_action_pressed("nav_right")
    var up = Input.is_action_pressed("nav_up")
    var down = Input.is_action_pressed("nav_down")
    if up:   pending_dir = "up"
    elif down: pending_dir = "down"
    elif right: pending_dir = "right"
    elif left: pending_dir = "left"
    # 如果都没有，保持原值（可能为空）
```

优先级保证了只要出现了 `up`，就会覆盖之前的任何方向。这与你的要求“最后一次出现的上/下为准”完全一致（因为每帧都会覆盖）。

### 3. 窗口内松开 L 键的处理
建议增加：窗口期间若 `L` 键被松开，立即中断窗口，不触发任何剑术，并切回普通状态。这样给了玩家“后悔药”。

### 4. 窗口长度初始值
你的 0.08 秒（约 5 帧）非常紧凑，建议初始值设为 `0.1` 秒（6 帧），让玩家有微小的宽松度，以后可根据测试逐步收紧。

### 5. “前/后”的方向转换
你文档中已经提到要根据 `facing_direction` 转换，实现时务必注意：

```gdscript
var is_facing_right = player.facing_direction > 0
match pending_dir:
    "up":    target_state = "SwordUppercutState"
    "down":  target_state = "SwordDownslashState"
    "right": target_state = "SwordDashState" if is_facing_right else "SwordSpinState"
    "left":  target_state = "SwordSpinState" if is_facing_right else "SwordDashState"
```

## 三、实现注意事项

- **窗口计时器**：可用 `Timer` 节点的 `one_shot`，或手动累加 `delta`。使用 `Timer` 更清晰。
- **状态退出**：窗口结束或中断时，记得 `stop_timer()`。
- **无缝衔接**：如果窗口内快速再次按下 `L`（连按），重置计时器，这样快速连按不会导致错误。
- **与其他状态冲突**：窗口期间如果玩家进入受伤、死亡等状态，应在 `exit()` 中取消窗口。
